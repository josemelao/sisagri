5) PLANO FUTURO DETALHADO (SEGURO E MODULAR)
================================================================
Fase F1 - Preparacao e especificacao de configuracoes
Objetivo:
- Definir contrato unico para preferencias do admin, com chave de ordenacao default por modulo/lista.

Entregaveis:
- Especificacao de chave de configuracao:
  - `default_sort_mode`: "id" | "nome"
  - (opcional futuro) `default_sort_field_por_modulo`: mapa por secao
- Regras de fallback:
  1. valor salvo no Supabase
  2. valor salvo localmente
  3. padrao seguro (nome, se usuario assim mantiver)

Validacao da fase:
- Documento atualizado com contrato e valores aceitos.
- Sem alteracao de comportamento em producao nesta fase.

Fase F2 - Persistencia de configuracoes (infra)
Objetivo:
- Criar tabela de configuracoes no Supabase sem quebrar fallback local.

Tabela proposta (sugestao conservadora):
- Nome: `admin_settings`
- Colunas minimas:
  - `id` (uuid/pk)
  - `scope` (text) -> ex.: "global"
  - `default_sort_mode` (text, valores: "id" | "nome")
  - `updated_at` (timestamp)
  - `updated_by` (text opcional, auditoria simples)

Regras de seguranca:
- Ler/escrever com anon key + politicas RLS adequadas ao fluxo admin existente.
- Nao usar service_role no frontend.
- Se Supabase falhar, manter operacao com storage local sem travar UI.

Validacao da fase:
- Teste de leitura da configuracao com fallback.
- Teste de escrita e reaplicacao apos reload.
- Teste de indisponibilidade do Supabase (graceful fallback).

Fase F3 - UI em Configuracoes do admin
Objetivo:
- Exibir opcao de ordenacao default de forma clara em Configuracoes.

Entregaveis de UI:
- Controle unico: "Ordenacao padrao das listas" com opcoes:
  - Nome (alfabetica)
  - ID (criacao)
- Acao salvar + feedback visual de sucesso/erro.
- Estado inicial da UI refletindo preferencia carregada.

Validacao da fase:
- Alterar para ID, salvar, recarregar, confirmar persistencia.
- Alterar para NOME, salvar, recarregar, confirmar persistencia.
- Simular erro remoto e confirmar fallback local.

Fase F4 - Integracao com render das listagens
Objetivo:
- Aplicar a configuracao salva no ponto central de ordenacao default das tabelas admin.

Regras:
- Mudanca minima: apenas ponto de ordenacao default.
- Nao quebrar busca, paginacao, filtros existentes.
- Nao alterar contratos de CRUD.

Validacao da fase:
- Smoke test em telas principais de listagem.
- Confirmar que busca e paginacao continuam estaveis.
- Confirmar que ordenacao default troca entre ID/NOME conforme configuracao.

Fase F5 - Futuro (nao bloqueante): ordenacao por cabecalho
Objetivo:
- Permitir clique nos cabecalhos para ordenar por coluna.

Diretrizes:
- Entrega posterior, separada, para reduzir risco.
- Reaproveitar o estado de ordenacao default como fallback inicial.
- Garantir acessibilidade minima (indicador visual de direcao ASC/DESC).

Validacao da fase:
- Clique em cabecalho ordena e alterna direcao.
- Integracao com busca + paginacao sem regressao.
