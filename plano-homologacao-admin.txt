Objetivo
- Implementar duas features de qualidade de vida no painel admin: FILTRO e PAGINACAO.
- Fazer isso de forma gradual e modular, com foco maximo em nao quebrar o codigo existente.
- Comecar pela pagina "Arquivos" (maior volume esperado). Depois replicar para as demais paginas de listagem.

Escopo (alvo)
- Paginas do admin que listam registros.
- Excecoes (nao entram no escopo): "Visao geral" e "Configuracoes".
- Primeira pagina: "Arquivos".

Regra mestra (operacao segura)
- Mudanca minima, no menor numero de arquivos possivel.
- Patch local em admin (sem refactor estrutural).
- Nao renomear IDs/classes/chaves existentes.
- Nao alterar contratos de DB.* sem necessidade explicita.
- Nao mudar fluxo de login, boot e inicializacao (dbInit e checks atuais).
- Nao expor rascunhos no app principal.

Economia maxima de token (para agentes)
- Trabalhar somente nos arquivos necessarios da etapa atual.
- Antes de mexer: localizar o ponto exato do render/handlers da pagina alvo.
- Evitar reler arquivos longos sem motivo. Se precisar, abrir so a secao relevante.
- Evitar "melhorias opportunisticas" fora do escopo (sem embelezar).

Definicao de pronto (DoD) por pagina
- Filtro:
  - Campo de busca filtra registros em tempo real (ou ao confirmar) sem quebrar o render.
  - Deve resetar a pagina para 1 ao mudar o filtro (para evitar pagina vazia).
  - Deve mostrar contagem (ex.: "X de Y") se isso ja existir de forma simples; se nao existir, nao inventar UI grande.
- Paginacao:
  - Controles para navegar (prev/next) e indicacao de pagina atual.
  - pageSize fixo por enquanto (ex.: 20/30), sem seletor inicialmente (menos risco).
  - Deve lidar com 0 registros sem erro.
- Compatibilidade:
  - CRUD existente segue funcionando (criar/editar/apagar).
  - Nada muda em DB.* e nenhuma chave persistida antiga e removida.

Design de implementacao (modular e reutilizavel)
- Objetivo tecnico: criar uma "camada" pequena e isolada para filtro/paginacao e reaproveitar nas paginas.
- Regra: SEM criar arquivo novo. Tudo em admin.js, em funcoes pequenas, com nomes novos (nao renomear existentes).
- Modelo recomendado (minimo):
  - Estado por pagina: { query, page, pageSize }.
  - Funcoes puras (sem depender do DOM):
    - normalizarTexto(s): string (lowercase + remover acentos se ja houver util no projeto; se nao houver, manter simples).
    - aplicarFiltro(items, query, getTextoDoItem): itemsFiltrados
    - paginar(items, page, pageSize): { pageItems, page, totalPages, totalItems }
  - Ligacao DOM por pagina (pequena):
    - ler inputs, atualizar estado, chamar render existente com a lista paginada.
    - handlers: oninput (filtro), onclick (prev/next).
- Persistencia de estado (opcional, depois de validado):
  - So adicionar localStorage se o admin ja usa padrao similar. Se nao, deixar em memoria para reduzir risco.

Padrao de UI (minimo, sem quebrar layout)
- Inserir um bloco pequeno acima da lista:
  - input de filtro (placeholder "Filtrar...").
  - texto curto de pagina ("Pagina X de Y") e botoes "Anterior/Proxima".
- IDs novos e especificos por pagina (nao colidir com IDs existentes).
- Nao mexer em CSS global a principio. So adicionar CSS se realmente necessario (e localizado).

Protocolo de validacao por etapa (obrigatorio)
- Ao finalizar cada etapa:
  - Rodar checklist manual descrito na etapa.
  - Entregar os comandos de commit.
  - Aguardar confirmacao explicita do usuario antes de marcar etapa como concluida e antes de atualizar historico.
